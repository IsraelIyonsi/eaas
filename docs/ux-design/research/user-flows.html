5Screen
Dashboard Overview
Main landing page showing system health and key metrics.

GET /health
GET /api/v1/logs?limit=10
  • Stat cards: Total Sent (today/7d/30d), Delivery Rate, Bounce Rate, Complaint Rate, Open Rate, Click Rate

  • System health indicators: API status, Worker status, Queue depth
  • Recent alerts panel: high bounce rate, complaint threshold warnings, domain verification failures

Review Health Status & Recent Emails

  • Hover stat cards for tooltip with trend comparison
  • Click any stat card to navigate to filtered analytics view
  • Click any email row to navigate to email detail in log viewer
  • Click alert to navigate to relevant management page

1Screen
Dashboard Overview
User notices the Bounce Rate stat card is highlighted red (above 2% threshold). An alert reads "Bounce rate at 4.2% - investigate recommended."
2Action
Click Bounce Rate Card or Alert
User clicks the bounce rate stat card or the alert link to investigate.
3Screen
Email Log Viewer (Pre-filtered: Bounced)
Navigates to the email logs page with status filter pre-set to "Bounced".
GET /api/v1/logs?status=bounced&sort=sent_at:desc
  • Table columns: Recipient, Subject, Sent At, Bounce Type, Bounce Reason
  • Status filter chip shows "Bounced" active
  • Date range defaults to last 7 days
  • Pagination: 50 rows per page
4Action
Click a Bounced Email Row
User clicks on a specific bounced email to see the full detail.
5Screen
Email Detail View
Full detail of the selected email including all status transitions and bounce diagnostics.
GET /api/v1/logs/{message_id}
  • Status timeline: Queued → Sent → Bounced (with timestamps)
  • Bounce reason: "550 5.1.1 The email account does not exist"
  • Bounce type: Hard Bounce (Permanent)
  • Recipient address, sender, subject, template used
  • Link to suppression entry (if auto-suppressed)
6Decision
Is This Address Already Suppressed?
User checks if the bounced address was auto-added to the suppression list.
Yes - Auto-Suppressed
7aSuccess
Suppression Confirmed
Address was automatically suppressed. No further sends will be attempted. User can continue reviewing other bounced emails.
No - Not Suppressed
7bAction
Navigate to Suppression List
User clicks "View Suppression List" link from the detail view to manually add the address.
8Screen
Suppression List Manager
User reviews the full suppression list, can search for the address, or manually add it.
GET /api/v1/suppressions
  • Search bar to find specific addresses
  • Table: Email, Reason, Suppressed At, Source Email
  • "Add Suppression" button for manual entries
  • Bulk select and remove option

1Screen
Template Manager - List View
User navigates to Templates via the sidebar. Sees all existing templates listed.
GET /api/v1/templates
  • Table: Name, Version, Send Count, Last Updated
  • "Create Template" primary action button
  • Search bar for filtering by name
  • Sort by created/updated date
2Action
Click "Create Template"
User clicks the primary action button to start creating a new template.
3Screen
Template Editor
Full-screen editor with split panes: code editor on left, live preview on right.
  • Template name field (required)
  • Subject template field with Liquid variable support
  • HTML body editor with syntax highlighting for Liquid/HTML
  • Optional plain text body editor
  • Variable schema editor: define expected variables and types
  • Preview panel (collapsed by default)
4Action
Write HTML Template with Variables
User writes the template HTML using Liquid syntax. Example: {{ customer_name }}, {{ invoice_total }}. Defines variable schema.
5Decision
Template Syntax Valid?
Editor performs real-time Liquid syntax validation as the user types.
No - Syntax Error
!Error
Syntax Validation Error
"Template syntax error at line 14: unexpected end of tag." Inline error markers in the editor.
Yes - Valid
6Action
Click "Preview" and Enter Sample Data
User expands the preview panel and enters sample variable values to see rendered output.
7Screen
Template Preview Panel
Live rendered preview of the template with the sample variables applied.
POST /api/v1/templates/{id}/preview
  • Rendered subject line displayed at top
  • Full HTML preview in iframe
  • Missing variables highlighted with warning markers
  • Sample variable input form auto-generated from schema
8Action
Click "Save Template"
User is satisfied with the preview and clicks Save.
9Decision
Save Successful?
System validates and persists the template.
No - Validation Failed
!Error
Save Failed
"Template name already exists" or "Template exceeds 512KB limit." User corrects and retries.
Yes - Saved
10Success
Template Saved Successfully
"Template 'Invoice Notification' created successfully." Toast notification. Redirected to template list.
POST /api/v1/templates
11Action
Navigate to API Documentation
User clicks "API Docs" in sidebar or copies the template_id from the success message to use in API calls.
  • API docs show how to send using template_id
  • Code example: POST /api/v1/emails/send with template_id and variables
  • Copy-ready cURL command

1Screen
Domain Manager - List View
User navigates to Domains via sidebar. Sees all registered domains with color-coded verification status.
GET /api/v1/domains
  • Table: Domain, Status (green/yellow/red badge), Last Verified, DNS Records
  • "Add Domain" primary action button
  • Status legend: Verified (green), Pending (yellow), Failed (red)
2Action
Click "Add Domain"
User clicks the primary action button. A modal dialog appears.
3Screen
Add Domain Modal
Modal with a single input field for the domain name.
  • Domain name input (e.g., "notifications.cashtrack.app")
  • Format hint: "Enter a domain or subdomain"
  • "Add Domain" submit button
  • "Cancel" secondary button
4Action
Enter Domain Name & Submit
User types the domain name and clicks "Add Domain".
5Decision
Domain Valid & Not Duplicate?
System validates format and checks for duplicates.
No - Invalid/Duplicate
!Error
Domain Rejected
"Invalid domain format" or "This domain is already registered." User corrects input.
Yes - Accepted
6Screen
DNS Records Display
System generates required DNS records via SES and displays them.
POST /api/v1/domains
7Screen
DNS Records Configuration Guide
All 5 DNS records displayed in a copyable format with instructions.
  • SPF TXT Record - with copy button
  • DKIM CNAME Record 1 - with copy button
  • DKIM CNAME Record 2 - with copy button
  • DKIM CNAME Record 3 - with copy button
  • DMARC TXT Record - with copy button
  • "Copy All" button for bulk copy
  • Instructions: "Add these records to your DNS provider, then click Verify."
8Action
Configure DNS Records (External)
User goes to their DNS provider (Cloudflare, Route53, etc.) and adds the records. This step happens outside the dashboard.
9Action
Click "Verify Now"
User returns to the dashboard and clicks the "Verify Now" button on the domain.
10Decision
All DNS Records Verified?
System checks SPF, DKIM, and DMARC records against expected values.
POST /api/v1/domains/{id}/verify
No - Partial/Failed
!Error
Verification Failed
Individual record results shown: SPF (verified), DKIM (failed - expected vs actual), DMARC (pending). "DNS propagation can take up to 48 hours. Try again later."
Yes - All Verified
11Success
Domain Verified
"Domain 'notifications.cashtrack.app' verified successfully. You can now send emails from this domain." Green status badge applied.

1Action
Navigate to Analytics
User clicks "Analytics" in the sidebar navigation.
2Screen
Analytics Dashboard
Full analytics view with charts and date range controls. Default view shows last 30 days.
GET /api/v1/analytics?group_by=day&range=30d
  • Date range selector with presets: Today, 7d, 30d, 90d, Custom
  • Time-series line chart: daily sends, deliveries, bounces, complaints
  • Donut chart: delivery status breakdown (Delivered, Bounced, Complained, Failed)
  • Bar chart: top sending templates by volume
  • Bar chart: top sending domains by volume
  • Table: per-API-key sending statistics
3Action
Set Custom Date Range
User selects "Custom" and sets a date range to compare two weeks of performance.
4Screen
Analytics - Filtered View
Charts refresh with the selected date range. User compares delivery rate vs bounce rate trends over the period.
GET /api/v1/analytics?from=2026-03-13&to=2026-03-27&group_by=day
5Action
Identify Problematic Template
User notices one template has a significantly higher bounce rate in the "Top Templates" bar chart. Clicks the template bar to drill down.
6Screen
Analytics - Template Drilldown
Filtered analytics view showing metrics for the selected template only.
GET /api/v1/analytics?template_id={id}&group_by=day
  • Template-specific delivery rate, bounce rate, open rate, click rate
  • Time-series chart for this template's performance
  • Link to view all email logs for this template
  • Link to edit this template
7Decision
Action Required?
User determines whether the template or recipient list needs attention.
Yes - Edit Template
8aAction
Click "Edit Template"
Navigates to the template editor (see Flow 3) to fix template issues.
Yes - View Logs
8bAction
Click "View Logs"
Navigates to email log viewer filtered by this template (see Flow 2).

Navigate to Suppressions

Suppression List Manager
Full suppression list with search, filter, and management controls.
GET /api/v1/suppressions

  • "Add Suppression" button
  • "Export CSV" button
  • Pagination controls
3Action
Search for an Address
User types a partial email address into the search bar. Results filter in real-time.
4Decision
Address Found in Suppression List?
System searches the suppression list for matching entries.
No - Not Found
5aScreen
Empty Search Results
"No suppressed addresses match your search." Table shows empty state.
Yes - Found
5bAction
Click "Remove" on the Entry
User clicks the remove button on the found suppression entry.
6Decision
Was This a Hard Bounce or Complaint?
System checks the suppression reason to determine if a warning is needed.
Yes - Warning Required
7aScreen
Confirmation Dialog with Warning
"This address was suppressed due to a hard bounce. Removing it will allow future sends, which may hurt your sender reputation. Are you sure?"
  • "Remove Anyway" danger button
  • "Cancel" secondary button
  • Original bounce reason and date shown
No - Manual Entry
7bSuccess
Suppression Removed
"Suppression removed for user@example.com. Emails can now be sent to this address."
DELETE /api/v1/suppressions/{email}
8Action
Click "Add Suppression"
User now wants to manually suppress a different address. Clicks "Add Suppression" button.
9Screen
Add Suppression Modal
Modal form for manually adding an email to the suppression list.
  • Email address input (required)
  • Reason dropdown: Manual, Known Invalid, User Request, Other
  • Optional notes field
  • "Add to Suppression List" submit button
10Action
Enter Address & Submit
User enters the email address, selects a reason, and clicks submit.
11Decision
Valid Email & Not Already Suppressed?
System validates the email format and checks for duplicates.

!Error
Suppression Failed
"Invalid email format" or "This address is already suppressed."
Yes - Added
12Success
Suppression Added
"spam@example.com added to suppression list. Future sends to this address will be blocked."
POST /api/v1/suppressions
