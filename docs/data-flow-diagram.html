✉ Flow 1: Send Single Email
The primary flow — from REST request to recipient inbox
POST /api/v1/emails/send
1 Client Application EXTERNAL
CashTrack or any tenant app sends an HTTP POST with the email payload and API key in the Authorization header.
POST /api/v1/emails/send
Authorization: Bearer eaas_live_abc123...
Content-Type: application/json

{
  "from": "billing@cashtrack.ng",
  "to": ["customer@gmail.com"],
  "cc": [],
  "bcc": [],
  "subject": "Invoice #1042",
  "html_body": "<h1>Hello...</h1>",
  "template_id": null,
  "attachments": [],
  "track_opens": true,
  "track_clicks": true,
  "idempotency_key": "inv-1042-send"
}
2 Nginx — TLS Termination + Rate Limiting GATEWAY
Terminates TLS (Let's Encrypt). Applies edge rate limiting. Routes to API server on port 8080.
// Nginx routes to upstream
proxy_pass http://eaas-api:8080;
3 ApiKeyAuthHandler — Authentication API
Extracts Bearer token, computes SHA-256 hash, and looks up the API key.
// Step 3a: Redis — check cached API key hash
GET cache:apikey:{sha256_hash}

// Step 3b: Cache miss → PostgreSQL fallback
SELECT * FROM api_keys WHERE key_hash = @hash AND status IN ('Active','Rotating')

// Step 3c: Set ClaimsPrincipal
Claims: { TenantId: guid, ApiKeyId: guid }
4 ValidationBehavior — FluentValidation API
MediatR pipeline behavior runs FluentValidation rules before the handler executes.
// Validates:
- from: required, valid email format
- to: required, 1..50 valid email addresses
- cc, bcc: optional, valid emails
- combined recipients (to + cc + bcc) ≤ 50
- subject: required (unless template_id)
- html_body or template_id: at least one required
5a SendEmailHandler — Rate Limit Check REDIS
Sliding window rate limiter using a Lua script. Max 100 sends/min per API key.
// Redis sorted set + Lua atomic check
Key: ratelimit:send:{apiKeyId}
Window: 60 seconds, Max: 100 requests
// If exceeded → 429 Too Many Requests
5b SendEmailHandler — Idempotency Check REDIS
If an idempotency key is provided, check Redis for a cached result. Returns the previous response if found.
Key: idempotency:{tenantId}:{key}
Value: { "id": "guid", "messageId": "eaas_xxx" }
TTL: 24 hours
5c SendEmailHandler — Verify Sender Domain POSTGRESQL
Extracts domain from the "from" address and verifies it is a verified domain for this tenant.
SELECT COUNT(*) FROM domains
WHERE tenant_id = @tenantId
  AND domain_name = 'cashtrack.ng'
  AND status = 'Verified'
  AND deleted_at IS NULL
5d SendEmailHandler — Suppression Check REDIS + PG
For EVERY recipient (to + cc + bcc), check the suppression list. Redis first, then PostgreSQL fallback.
// For each recipient:
1. Redis: GET suppression:{tenantId}:{email}
2. [Cache miss] PostgreSQL:
  SELECT 1 FROM suppression_list
  WHERE tenant_id = @tid AND email_address = @email

// If any recipient suppressed → 400 error
5e SendEmailHandler — Persist Email + Event POSTGRESQL
INSERT the email record with status=Queued and the initial Queued event. Single SaveChanges transaction.
INSERT INTO emails (
  id, tenant_id, api_key_id, message_id,
  from_email, to_emails, cc_emails, bcc_emails,
  subject, html_body, text_body,
  template_id, variables, tags, metadata,
  status='Queued', created_at
)

INSERT INTO email_events (
  id, email_id, event_type='Queued', data='{}', created_at
)
5f SendEmailHandler — Publish to Queue RABBITMQ
Publishes a SendEmailMessage via MassTransit to the RabbitMQ exchange. The message lands in the eaas-emails-send queue.
Queue: eaas-emails-send
Message: SendEmailMessage {
  EmailId: guid,
  TenantId: guid,
  From: "billing@cashtrack.ng",
  To: '["customer@gmail.com"]',
  Subject, HtmlBody, TextBody, TemplateId,
  Variables, Tags, Metadata, TrackOpens, TrackClicks
}
5g SendEmailHandler — Store Idempotency Key REDIS
Stores the idempotency key with a 24-hour TTL so duplicate requests return the same response.
SET idempotency:{tenantId}:{key}
Value: { "id": "guid", "messageId": "eaas_xxx" }
TTL: 86400 seconds (24h)
5h Return 202 Accepted API
The API immediately returns. Email delivery happens asynchronously via the worker.
HTTP/1.1 202 Accepted
{
  "id": "a1b2c3d4-...",
  "messageId": "eaas_a1b2c3d4...",
  "status": "queued"
}
6 RabbitMQ — Message Waiting RABBITMQ
Message sits in eaas-emails-send queue until a worker consumer picks it up. Durable classic queue with prefetch configured.
7a SendEmailConsumer — Load Email POSTGRESQL
Worker loads the full Email entity from PostgreSQL by EmailId.
SELECT * FROM emails WHERE id = @emailId
7b SendEmailConsumer — Load + Render Template WORKER
If template_id is set, load the template and render via Fluid (Liquid engine) with provided variables.
// Load template
SELECT * FROM templates WHERE id = @templateId AND deleted_at IS NULL

// Fluid renders with variables
Input: subject_template + html_body + text_body + { name: "Tunde", invoice_url: "..." }
Output: rendered subject, html, text
7c TrackingPixelInjector — Open Tracking WORKER
If trackOpens=true, injects a 1x1 transparent pixel <img> tag before </body>.
// Injected HTML
<img src="https://BASE_URL/track/open/{hmac_token}"
  width="1" height="1" style="display:none" />
7d ClickTrackingLinkRewriter — Click Tracking PG + WORKER
If trackClicks=true, rewrites every <a href> to a tracking redirect URL. Stores original URLs in PostgreSQL.
// For each link in HTML:
Original: <a href="https://pay.cashtrack.ng/inv/1042">
Rewritten: <a href="https://BASE_URL/track/click/{token}">

INSERT INTO tracking_links (id, email_id, token, original_url, created_at)
7e Update Email Status → Sending POSTGRESQL
Updates the email record status to Sending and inserts a Sent event.
UPDATE emails SET status = 'Sending' WHERE id = @id
INSERT INTO email_events (event_type='Sent')
7f AWS SES — Send Email AWS SES
Depending on attachments, sends via SES v2 Simple or Raw (MimeKit-built MIME message).
// No attachments:
SES v2 SendEmail (Simple: from, to, cc, bcc, subject, html, text)

// With attachments:
MimeKit builds multipart/mixed MIME → SES v2 SendRawEmail(stream)

// Returns:
{ Success: true, MessageId: "ses-msg-id" }
7g Persist SES Result POSTGRESQL
Stores the SES Message ID and SentAt timestamp on the email record.
UPDATE emails SET
  ses_message_id = 'ses-msg-id',
  sent_at = NOW()
WHERE id = @id
8 SMTP Delivery to Recipient AWS SES
AWS SES delivers the email via SMTP to the recipient's mail server (Gmail, Outlook, Yahoo, etc.).
SES SMTP → mx.google.com → customer@gmail.com
// Delivery events published to SNS (see Flow 3)
9 Recipient Inbox EXTERNAL
Email appears in the recipient's inbox. Opens and clicks are tracked via embedded pixel and rewritten links (see Flows 4 and 5).
📦 Flow 2: Batch Send
Send up to 100 emails in a single API call
POST /api/v1/emails/batch
1 Client Application EXTERNAL
Sends an array of up to 100 email payloads in a single request.
POST /api/v1/emails/batch
Authorization: Bearer eaas_live_abc123...

{
  "emails": [
    { "from": "...", "to": [...], "subject": "...", ... },
    // ... up to 100 items
  ]
}
2 Nginx + Auth + Validation API
Same TLS, authentication, and validation pipeline as single send. Validates the batch array length ≤ 100.
3 SendBatchHandler — Generate Batch ID API
Generates a unique batch_id to group all emails in this batch.
batch_id = Guid.NewGuid()
4 SendBatchHandler — For Each Email API
Iterates through each email in the batch. For each one: validates, checks suppression, creates DB record, publishes to queue.
// For each of N emails (max 100):
1. Validate from, to, subject, body
2. Check suppression list (Redis → PG)
3. INSERT INTO emails (status='Queued', batch_id)
4. INSERT INTO email_events (event_type='Queued')
5. Publish SendEmailMessage to RabbitMQ
5 RabbitMQ — N Individual Messages RABBITMQ
Each email becomes its own independent message on the eaas-emails-send queue. Workers process them in parallel.
Queue: eaas-emails-send
Messages: N individual SendEmailMessage (1 per email in batch)
// Each processed independently by SendEmailConsumer
6 Return 202 Accepted API
Returns the batch ID and per-email results immediately.
HTTP/1.1 202 Accepted
{
  "batchId": "batch-guid",
  "results": [
    { "messageId": "eaas_xxx", "status": "queued" },
    { "messageId": "eaas_yyy", "status": "queued" },
    // ...
  ]
}
⚠ Flow 3: Bounce, Complaint & Delivery Handling
SNS webhook processing for SES delivery events
POST /webhooks/sns
1 Recipient Mail Server Rejects / Complains EXTERNAL
The recipient's mail server rejects the email (bounce) or the user marks it as spam (complaint). Successful delivery also triggers a delivery event.
2 AWS SES Detects Event AWS SES
SES detects the bounce, complaint, or delivery event and publishes it to the configured SNS topic.
3 AWS SNS Notification AWS SNS
SNS sends an HTTP POST to the webhook endpoint with the event wrapped in an SNS envelope.
POST /webhooks/sns
Content-Type: text/plain

{
  "Type": "Notification",
  "MessageId": "sns-msg-id",
  "TopicArn": "arn:aws:sns:...",
  "Message": "{ SES notification JSON }",
  "SigningCertURL": "https://sns.*.amazonaws.com/...",
  "Signature": "base64..."
}
4 SnsMessageHandler — Validate + Route WEBHOOK
Validates the SNS message signature by checking the cert URL is *.amazonaws.com. Routes by message type.
// Validate cert URL domain
Assert: SigningCertURL host ends with .amazonaws.com

// Route by Type:
"SubscriptionConfirmation" → GET SubscribeURL (auto-confirm)
"Notification" → Parse SES notification → route to handler
BOUNCE PATH (bounceType detected)
5a BounceHandler — Lookup Email POSTGRESQL
Finds the email record by the SES Message ID from the notification.
SELECT * FROM emails
WHERE ses_message_id = @notification.Mail.MessageId
Permanent (Hard) Bounce
1. INSERT into suppression_list (reason: HardBounce)
2. Redis: add to suppression cache
3. UPDATE emails SET status = Bounced
4. INSERT email_event (Bounced) with diagnostic data
Transient (Soft) Bounce
1. INSERT email_event (Bounced, transient=true)
2. No suppression — may retry later
3. Email status unchanged
COMPLAINT PATH (complaint detected)
5b ComplaintHandler WEBHOOK
User marked the email as spam. Auto-suppresses the recipient immediately.
1. INSERT INTO suppression_list (reason: Complaint)
2. Redis: suppression:{tenantId}:{email} = true
3. UPDATE emails SET status = 'Complained'
4. INSERT INTO email_events (event_type='Complained')
DELIVERY PATH (delivery confirmed)
5c DeliveryHandler WEBHOOK
SES confirms the email was accepted by the recipient's mail server.
1. UPDATE emails SET status = 'Delivered', delivered_at = NOW()
2. INSERT INTO email_events (event_type='Delivered')
👁 Flow 4: Open Tracking
Tracking pixel loaded when a recipient opens the email
GET /track/open/{token}
1 Recipient Opens Email EXTERNAL
The recipient opens the email in their mail client (Gmail, Outlook, Apple Mail, etc.). The client loads remote images.
2 Browser Loads Tracking Pixel EXTERNAL
The mail client sends a GET request for the embedded 1x1 pixel image.
GET https://BASE_URL/track/open/{hmac_token}
User-Agent: Mozilla/5.0 (iPhone; ...)
X-Forwarded-For: 102.89.xx.xx
3 ITrackingTokenService — Validate HMAC Token WEBHOOK
Validates the HMAC token signature and extracts the emailId from it.
Token: {hmac_token}
→ HMAC-SHA256 validate with secret key
→ Extract: emailId = guid
4 Record Open Event POSTGRESQL
Inserts an Opened event and sets opened_at on first open only.
INSERT INTO email_events (
  email_id, event_type='Opened',
  data: { user_agent: "...", ip: "102.89.xx.xx" }
)

// First open only:
UPDATE emails SET opened_at = NOW()
WHERE id = @id AND opened_at IS NULL
5 Return 1x1 Transparent GIF WEBHOOK
Returns a minimal transparent GIF with no-cache headers so each open is tracked.
HTTP/1.1 200 OK
Content-Type: image/gif
Content-Length: 43
Cache-Control: no-store, no-cache, must-revalidate
Pragma: no-cache

// 43-byte transparent 1x1 GIF binary
🔗 Flow 5: Click Tracking
Tracking redirect when a recipient clicks a link in the email
GET /track/click/{token}
1 Recipient Clicks a Link EXTERNAL
The recipient clicks a rewritten link in the email body.
// What the recipient sees in the email:
<a href="https://BASE_URL/track/click/{token}">Pay Invoice</a>
2 Lookup Original URL POSTGRESQL
Looks up the tracking_links table by token to find the original destination URL.
SELECT original_url, email_id FROM tracking_links
WHERE token = @token

Result: https://pay.cashtrack.ng/inv/1042
3 Record Click Event POSTGRESQL
Inserts a Clicked event and updates the tracking_links record with the click timestamp.
INSERT INTO email_events (
  email_id, event_type='Clicked',
  data: { original_url: "https://pay.cashtrack.ng/inv/1042" }
)

UPDATE tracking_links SET clicked_at = NOW()
WHERE token = @token
4 302 Redirect to Original URL WEBHOOK
Sends a 302 redirect to the original destination URL. The recipient is seamlessly forwarded.
HTTP/1.1 302 Found
Location: https://pay.cashtrack.ng/inv/1042
🔑 Flow 6: API Key Rotation
Zero-downtime key rotation with a 24-hour grace period
POST /api/v1/keys/{id}/rotate
1 Admin Requests Rotation EXTERNAL
Admin sends a POST to rotate an existing API key.
POST /api/v1/keys/{keyId}/rotate
Authorization: Bearer eaas_live_current_key...
2 RotateApiKeyHandler — Generate New Key API
Generates a new cryptographically random API key with the eaas_live_ prefix.
new_key = "eaas_live_" + RandomBytes(32).ToBase64Url()
3 PostgreSQL — Update Old + Insert New POSTGRESQL
Marks the old key as Rotating with a 24-hour expiry. Inserts the new key as Active.
// Old key:
UPDATE api_keys SET
  status = 'Rotating',
  rotating_expires_at = NOW() + INTERVAL '24 hours',
  replaced_by_id = @newKeyId
WHERE id = @oldKeyId

// New key:
INSERT INTO api_keys (
  id, tenant_id, key_hash, status='Active', created_at
)
4 Redis — Invalidate Old Key Cache REDIS
Removes the old key from the Redis cache so the auth handler fetches the updated status from PostgreSQL.
DEL cache:apikey:{old_key_hash}
5 Return New Plaintext Key API
Returns the new key in plaintext (shown only once, never stored in plaintext).
HTTP/1.1 200 OK
{
  "newKey": "eaas_live_abc123newkey...",
  "oldKeyExpiresAt": "2026-03-28T12:00:00Z",
  "message": "Old key valid for 24 more hours"
}
6 Grace Period Behavior (24 hours) API
During the grace period, the ApiKeyAuthHandler accepts both the old key (status=Rotating, not yet expired) and the new key (status=Active). After 24 hours, the old key's rotating_expires_at is in the past, so auth rejects it.
// Auth handler logic:
if (key.Status == "Rotating")
  if (key.RotatingExpiresAt > DateTime.UtcNow)
    → ALLOW (grace period active)
  else
    → DENY (expired)
📄 Flow 7: Template Rendering
Fluid (Liquid) template engine rendering with Redis caching
1 SendEmailConsumer Receives Message with template_id WORKER
The consumer receives a message from RabbitMQ that specifies a template_id instead of inline HTML.
message.TemplateId = guid
message.Variables = '{ "name": "Tunde", "invoice_url": "https://..." }'
2 Redis — Check Template Cache REDIS
Attempts to load a cached version of the compiled template from Redis.
GET template:{templateId}
Cache Hit
Use the cached template directly. Skip database query.
Cache Miss
Load from PostgreSQL:
SELECT * FROM templates WHERE id = @id AND deleted_at IS NULL
Then cache the result in Redis for future use.
3 Fluid (Liquid Engine) — Parse + Render WORKER
Fluid parses the subject template, HTML body, and text body. Injects the provided variables using Liquid syntax.
// Input templates:
subject_template: "Invoice for {{name}}"
html_body: "<h1>Hi {{name}}</h1><a href='{{invoice_url}}'>Pay</a>"
text_body: "Hi {{name}}, pay at {{invoice_url}}"

// Variables:
{ name: "Tunde Adebayo", invoice_url: "https://pay.cashtrack.ng/inv/1042" }

// Output:
subject: "Invoice for Tunde Adebayo"
html: "<h1>Hi Tunde Adebayo</h1><a href='https://pay...'>Pay</a>"
text: "Hi Tunde Adebayo, pay at https://pay..."
4 Return Rendered Content WORKER
Rendered subject, HTML body, and text body are passed to the next steps in the send flow (tracking injection, SES delivery).
RenderedTemplate {
  Subject: "Invoice for Tunde Adebayo",
  HtmlBody: "<h1>Hi Tunde Adebayo...",
  TextBody: "Hi Tunde Adebayo, pay at..."
}
🔄 Flow 8: Retry & Dead Letter Queue
MassTransit retry policy and error handling
1 SendEmailConsumer Throws Exception WORKER
The consumer fails to process the message (SES error, database timeout, template not found, etc.). The exception is caught by MassTransit.
// Example failures:
- SES delivery failed (throttled, invalid address)
- PostgreSQL connection timeout
- Template not found (status → Failed, no retry)
- Unexpected exception
2 Retry 1 — After 1 Second MASSTRANSIT
MassTransit retries the message after a 1-second delay.
Delay: 1 second
// Success? → Done. Failure? → Retry 2
3 Retry 2 — After 5 Seconds MASSTRANSIT
Second retry attempt after a 5-second delay.
Delay: 5 seconds
// Success? → Done. Failure? → Retry 3
4 Retry 3 — After 30 Seconds MASSTRANSIT
Final retry attempt after a 30-second delay.
Delay: 30 seconds
// Success? → Done. Failure? → Dead Letter Queue
5 Dead Letter Queue RABBITMQ
After all 3 retries fail, MassTransit moves the message to the error queue. The email status is set to Failed.
Queue: eaas-emails-send_error (Dead Letter Queue)

// PostgreSQL update:
UPDATE emails SET status = 'Failed',
  error_message = 'SES delivery failed after 3 retries'
WHERE id = @id

INSERT INTO email_events (event_type='Failed', data='{ "error": "..." }')
6 Manual Inspection Required ADMIN
Messages in the DLQ require manual inspection via the RabbitMQ management UI or Dashboard. Admins can investigate the failure, fix the root cause, and optionally republish the message.
// RabbitMQ Management UI:
http://localhost:15672 → Queues → eaas-emails-send_error
→ Get Message(s) → Inspect payload → Fix & Republish
