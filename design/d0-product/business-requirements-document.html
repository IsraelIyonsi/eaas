Section 01

Executive Summary

EaaS currently serves as an outbound-only transactional email API. This Business Requirements Document outlines the business case for adding inbound email capabilities — enabling customers to receive, parse, route, and process incoming emails through EaaS.

The global email API market is valued at $3.1–4.2B (2024) with projected growth to $8.2–16.2B by 2033. Inbound email is a chronically underserved segment — most providers (SendGrid, Postmark, Mailgun) treat it as a secondary feature with stateless webhooks, no persistent storage, and no conversation threading. The emerging AI agent category (AgentMail, $6M seed) validates persistent inbox infrastructure as a high-growth opportunity.

Section 04

Problem Statement

Current state: EaaS is outbound-only. Customers who need to process inbound emails must resort to suboptimal workarounds:

  • Use a separate provider (SendGrid Inbound Parse, Postmark Inbound) — adding cost, complexity, and vendor management overhead
  • Build custom SMTP receiving infrastructure — significant engineering overhead, requiring weeks of development and ongoing maintenance
  • Lose conversation context — no thread linking between sent and received emails, breaking the user experience

Section 05

Proposed Solution — High Level

EaaS will add comprehensive inbound email capabilities, built on the existing AWS SES infrastructure to minimize vendor complexity and leverage established operational knowledge.

Capability Description Technology
Email Reception Receive inbound emails via AWS SES inbound receiving (already on SES for outbound — single vendor) AWS SES Inbound + SNS
MIME Parsing Automatic parsing of headers, body (text + HTML), and attachments MimeKit (.NET)
Persistent Storage Raw MIME stored in S3, parsed metadata in PostgreSQL (not just stateless webhooks) AWS S3 + PostgreSQL
Routing Rules Configurable rules to route inbound emails to webhooks, forward to addresses, or store silently Rules engine + background workers
Thread Linking Automatic reply tracking via In-Reply-To and References headers Message-ID correlation
Dashboard Full inbox viewer with email list, detail view, thread view, and analytics Next.js + shadcn/ui
Developer API REST endpoints for querying, filtering, downloading inbound emails and attachments .NET 8 Web API
Webhook Dispatch Real-time notifications to customer endpoints when emails arrive, with HMAC signing Background workers + retry queue

Section 06

Business Requirements

Revenue & Pricing

  • BR-001 Inbound email processing MUST be included in paid plans at no additional per-email cost (competitive with Mailgun's free inbound model)
  • BR-002 Free tier MUST include inbound capabilities (up to plan limits) to drive adoption and reduce friction for new customers
  • BR-003 Storage retention MUST be configurable (30/60/90 days) with premium tiers for longer retention periods

Customer Experience

  • BR-004 Customers MUST be able to receive, view, and search inbound emails from the dashboard with full-text search capabilities
  • BR-005 Customers MUST be able to configure routing rules without engineering support via a self-service UI
  • BR-006 Reply tracking MUST automatically link inbound replies to original outbound emails to maintain conversation threads
  • BR-007 Webhook delivery MUST include full parsed email data in JSON format (not multipart/form-data) for easy integration
  • BR-008 Setup onboarding MUST guide users through MX record configuration with verification polling and provider-specific instructions

Operational

  • BR-009 Inbound processing MUST handle emails up to 30MB (attachments included) without degradation
  • BR-010 Webhook delivery MUST retry failed deliveries with exponential backoff for at least 3 days before moving to dead letter queue
  • BR-011 System MUST process inbound emails within 30 seconds of receipt (P95 latency target)
  • BR-012 Dashboard MUST show real-time inbound email activity with live updates (no manual refresh required)
  • BR-013 API documentation MUST include quickstart guide, full payload reference, and SDK examples in at least 3 languages

Compliance & Security

  • BR-014 All inbound email data MUST be encrypted at rest (S3 AES-256, PostgreSQL TDE) and in transit (TLS 1.2+)
  • BR-015 Customers MUST be able to delete inbound emails and all associated data on demand (GDPR Article 17 right to erasure)
  • BR-016 Spam and virus verdicts from SES MUST be surfaced to customers for filtering decisions in both API and dashboard
  • BR-017 Webhook payloads MUST be signed with HMAC-SHA256 for verification, with signing secret rotation support

Section 08

Scope & Boundaries

In Scope

  • Inbound email reception via AWS SES
  • MIME parsing (headers, body, attachments)
  • Persistent storage (S3 + PostgreSQL)
  • Routing rules (webhook, forward, store)
  • Reply tracking (In-Reply-To matching)
  • Dashboard (email list, detail, thread view, rules management, analytics)
  • REST API (list, get, raw download, attachment download, delete)
  • Webhook dispatch with HMAC signing
  • Developer documentation (API docs, webhook guide, setup guide)
  • Onboarding wizard (MX record setup, domain verification)

Out of Scope (Future Phases)

  • IMAP/POP3 access
  • AI-powered email classification / extraction
  • MCP server integration for AI agents
  • Edge processing (Cloudflare Workers)
  • Self-hosted deployment option
  • Custom SMTP receiving (non-SES)
  • Email forwarding chain (forward inbound to another email address) — Sprint 5

Section 09

Risk Assessment

Risk Probability Impact Mitigation Strategy
AWS SES regional limitations — only 3 regions support inbound receiving Medium High Start with eu-west-1 (Ireland), expand as customer demand requires. Monitor SES roadmap for new regions.
Large email attachments causing processing delays and storage costs Medium Medium Stream-based processing, 30MB hard limit, async pipeline with backpressure handling.
Spam/abuse of inbound endpoints driving infrastructure costs High Medium Built-in SES spam/virus scanning, configurable thresholds, rate limiting per domain, automated quarantine.
Customer confusion with MX record setup leading to support burden High Medium Step-by-step wizard with provider-specific guides (GoDaddy, Cloudflare, Route53), auto-verification polling.
Webhook delivery failures at scale impacting customer trust Medium High Exponential retry (3 days), dead letter queue, manual retry from dashboard, delivery health monitoring.
GDPR data retention compliance risk for EU customers Medium High Configurable retention policies, automated S3 lifecycle cleanup, deletion API, audit trail.
Competitor launches better inbound before EaaS ships Low Medium Fast execution (3 sprints / 5 weeks), focus on persistent inbox differentiator that competitors lack.
