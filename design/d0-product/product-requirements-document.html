Section 01

Product Vision

"Make EaaS the only email API developers need — send, receive, and manage email conversations through a single, elegant platform."

EaaS inbound email transforms our product from a send-only API into a complete email platform. Developers can receive emails, parse content, route to webhooks, track conversation threads, and monitor everything from a modern dashboard — all without managing SMTP infrastructure.

Today, developers who need both outbound and inbound email are forced to stitch together multiple vendors: one for sending (SendGrid, Postmark) and another for receiving (Mailgun, AWS SES raw). This creates fragmented billing, no thread linking between sent and received messages, inconsistent APIs, and operational overhead managing two systems.

EaaS inbound email eliminates this fragmentation. With a single API key, a single dashboard, and a single billing relationship, developers get send + receive + thread linking + persistent inbox + a modern management UI. No other provider offers this combination today.

Positioning: Developer-friendly email platform with persistent inbox, conversation threading, and modern dashboard — capabilities only Resend partially offers today. EaaS will be the first platform to natively link outbound and inbound emails into conversation threads, giving developers conversation-level context that no competitor provides out of the box.

User Stories with Acceptance Criteria

Epic 1: Inbound Email Reception
US-4.1
As Alex, I want EaaS to receive emails sent to my domain, so that I can process customer replies in my app.
  • Emails sent to configured domain are received by EaaS within 30 seconds
  • Raw MIME is stored in S3 for forensic access
  • Parsed metadata (from, to, subject, body, headers) is stored in PostgreSQL
  • Email appears in dashboard within 60 seconds of receipt
  • SPF, DKIM, DMARC, spam, and virus verdicts are captured
US-4.2
As Alex, I want inbound emails automatically parsed into structured JSON, so that I don't have to deal with MIME parsing.
  • HTML body and text body are extracted separately
  • All headers are parsed into key-value pairs
  • In-Reply-To and References headers are extracted for threading
  • Attachments are extracted with filename, content type, size, and download URL
  • Inline images (CID) are identified separately from file attachments
US-4.3
As Alex, I want replies to my outbound emails automatically linked, so that I can show conversation threads.
  • In-Reply-To header is matched against outbound email Message-IDs
  • Matching inbound email has outbound_email_id set
  • API response includes the linked outbound email ID
  • Thread view in dashboard shows the full conversation chain
Epic 2: Routing Rules
US-4.4
As Sarah, I want to create rules that route emails to different webhooks based on the recipient address, so that support@ goes to our helpdesk and billing@ goes to our accounting system.
  • Rules can match on recipient pattern (exact, wildcard, prefix like "support@")
  • Rules support 3 actions: webhook, forward, store
  • Rules have configurable priority (lower number = higher priority)
  • Multiple rules can be active simultaneously
  • Rules are scoped to a specific verified domain
US-4.5
As Sarah, I want to manage routing rules from the dashboard without writing code, so that I can update routing without bothering developers.
  • Dashboard provides create/edit/delete UI for rules
  • Rule form validates webhook URLs and email addresses
  • Rules can be enabled/disabled with a toggle
  • Priority can be reordered
  • Changes take effect within 60 seconds
US-4.6
As Alex, I want a CRUD API for managing inbound rules programmatically, so that I can automate rule management in my deployment pipeline.
  • POST /api/v1/inbound/rules — create rule (returns 201)
  • GET /api/v1/inbound/rules — list rules with pagination
  • GET /api/v1/inbound/rules/{id} — get rule detail
  • PUT /api/v1/inbound/rules/{id} — update rule
  • DELETE /api/v1/inbound/rules/{id} — delete rule (returns 204)
  • All endpoints require authentication and enforce tenant isolation
Epic 3: Webhook Dispatch
US-4.7
As Alex, I want to receive a webhook when an email arrives, so that my app can process it in real-time.
  • Webhook POST contains full parsed email data in JSON (not multipart/form-data)
  • Webhook includes HMAC-SHA256 signature in X-EaaS-Signature header
  • Webhook includes timestamp for replay protection
  • Payload includes: id, messageId, from, to, cc, subject, textBody, htmlBody, headers, attachments (with download URLs), verdicts, receivedAt, inReplyTo, outboundEmailId
  • Webhook is dispatched within 30 seconds of email receipt
US-4.8
As Alex, I want failed webhooks to be retried automatically, so that I don't lose emails during temporary outages.
  • Failed webhooks (5xx, timeout) are retried with exponential backoff
  • Retry schedule: 1min, 5min, 30min, 2hr, 8hr, 24hr, 48hr, 72hr
  • After all retries exhausted, email moves to dead letter queue
  • Webhook delivery attempts are logged with status codes and response times
  • Manual retry available from dashboard
Epic 4: Inbound Email API
US-4.9
As Alex, I want to query received emails via REST API, so that I can build custom views in my app.
  • GET /api/v1/inbound/emails — paginated list with filters (from, to, dateFrom, dateTo, status, hasAttachments)

US-4.10
As Marcus, I want to search inbound emails by content, so that my AI agents can find relevant conversations.
  • Full-text search across subject, from_email, to_emails

API Contract Summary

All endpoints require Authorization: Bearer <api-key> header. Responses follow the standard EaaS envelope: { "data": ..., "meta": { "page", "pageSize", "total" } }. All IDs are UUIDs. Timestamps are ISO 8601 UTC.

Inbound API Endpoints

Method Path Description Sprint
POST /api/v1/inbound/rules Create routing rule Sprint 4
GET /api/v1/inbound/rules List rules (paginated) Sprint 4
GET /api/v1/inbound/rules/{id} Get rule by ID Sprint 4
PUT /api/v1/inbound/rules/{id} Update rule Sprint 4
DELETE /api/v1/inbound/rules/{id} Delete rule Sprint 4
GET /api/v1/inbound/emails List inbound emails (paginated, filtered) Sprint 5
GET /api/v1/inbound/emails/{id} Get email detail with attachments Sprint 5
GET /api/v1/inbound/emails/{id}/raw Download raw MIME (S3 presigned URL) Sprint 5
GET /api/v1/inbound/emails/{id}/attachments/{attId} Download attachment Sprint 5
DELETE /api/v1/inbound/emails/{id} Delete inbound email Sprint 5

Internal Endpoint

Method Path Description Sprint
POST /webhooks/sns/inbound SNS inbound notification (internal — not exposed to customers) Sprint 4
Note
The SNS inbound endpoint handles AWS SNS subscription confirmation and notification messages. It is not part of the public API and is secured via SNS message signature verification, not API key authentication.
Section 07

Webhook Payload Schema

When an inbound email is received and a routing rule matches with a webhook action, EaaS dispatches an HTTP POST to the configured URL with the following JSON payload. The request includes an X-EaaS-Signature header containing an HMAC-SHA256 signature for verification, and an X-EaaS-Timestamp header for replay protection.

{
  "event": "email.received",
  "data": {
    "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "messageId": "<CABx28Z=abc123@mail.gmail.com>",
    "from": {
      "email": "sender@example.com",
      "name": "John Doe"
    },
    "to": [
      {
        "email": "support@customer.com",
        "name": ""
      }
    ],
    "cc": [],
    "subject": "Re: Order #12345",
    "textBody": "Thanks for the update. I'll review the invoice and get back to you.",
    "htmlBody": "<p>Thanks for the update. I'll review the invoice and get back to you.</p>",
    "headers": {
      "In-Reply-To": "<original-message-id@eaas.dev>",
      "References": "<original-message-id@eaas.dev>",
      "X-Mailer": "Apple Mail"
    },
    "attachments": [
      {
        "id": "f7e8d9c0-b1a2-3456-7890-abcdef123456",
        "filename": "invoice.pdf",
        "contentType": "application/pdf",
        "sizeBytes": 45000,
        "downloadUrl": "/api/v1/inbound/emails/{id}/attachments/{attId}"
      }
    ],
    "verdicts": {
      "spam": "pass",
      "virus": "pass",
      "spf": "pass",
      "dkim": "pass",
      "dmarc": "pass"
    },
    "receivedAt": "2026-04-01T12:00:00Z",
    "inReplyTo": "<original-message-id@eaas.dev>",
    "outboundEmailId": "b2c3d4e5-f6a7-8901-bcde-f12345678901"
  },
  "timestamp": "2026-04-01T12:00:01Z"
}
Signature Verification
Compute HMAC-SHA256(webhook_secret, timestamp + "." + raw_body) and compare with the X-EaaS-Signature header value. Reject requests where X-EaaS-Timestamp is more than 5 minutes old to prevent replay attacks.

Payload Field Reference

Field Type Description
event string Always "email.received" for inbound emails
data.id UUID Unique EaaS identifier for this inbound email
data.messageId string RFC 2822 Message-ID header from the email
data.from object Sender with email and name fields
data.to array Array of recipient objects with email and name
data.cc array Array of CC recipient objects (may be empty)
data.subject string Email subject line
data.textBody string|null Plain text body (null if not present)
data.htmlBody string|null HTML body (null if not present)
data.headers object All email headers as key-value pairs
data.attachments array Attachment metadata with download URLs (not base64 content)
data.verdicts object Security verdicts: spam, virus, spf, dkim, dmarc — each "pass" or "fail"
data.receivedAt ISO 8601 Timestamp when EaaS received the email
data.inReplyTo string|null In-Reply-To header value (null if not a reply)
data.outboundEmailId UUID|null Linked outbound email ID if this is a reply to a sent email
timestamp ISO 8601 Timestamp when the webhook was dispatched
Section 08

Design Requirements

Screens to Design

The following 10 screens must be designed as part of the inbound email feature. Each screen requires high-fidelity mockups in Figma, delivered as interactive prototypes for engineering handoff.

  1. Inbound Email List — Paginated table with filters, search, real-time updates, and bulk actions. Columns: Status, From, To, Subject, Received, Attachments, Webhook Status.
  2. Inbound Email Detail — Full email view with tabs for HTML preview (rendered in sandboxed iframe), plain text, raw headers, and JSON payload. Attachment list with download buttons. Security verdict badges.
  3. Thread View — Conversation chain linking sent and received emails in chronological order. Direction indicators (sent/received), expandable message previews, and click-through to full detail.
  4. Inbound Rules Management — List view of all routing rules with enable/disable toggles, drag-to-reorder priority, and inline create/edit forms. Rule form includes pattern matcher, action selector, and webhook URL input.
  5. Inbound Setup Wizard — 5-step guided onboarding flow: domain selection, MX record display with copy buttons, auto-verification polling, webhook configuration, and test email verification. Provider-specific DNS guides.
  6. Inbound Analytics Dashboard — Metric cards (Total Received, Processed, Failed, Avg Processing Time), volume chart (hourly/daily/weekly), status breakdown donut chart, top senders table, and time range selector.
  7. Notification Preferences — Alert configuration per domain: volume spike, processing failures, spam threshold. Channel selection (webhook, email) with threshold inputs.
  8. API Documentation Page — Endpoint reference with request/response examples, authentication guide, error codes, and interactive API explorer.
  9. Webhook Integration Guide — Code snippets in 4 languages (Node.js, Python, C#, Go), HMAC verification walkthrough, payload schema reference, and testing tool recommendations.
  10. Inbound Setup Guide — Step-by-step instructions with architecture diagram showing the full inbound flow (SES → S3 → SNS → EaaS → Webhook). FAQ section addressing common issues.

Design Constraints

Design System: Must use existing EaaS design system (Next.js + shadcn/ui + Tailwind CSS)
Dark Mode: Must support dark mode with proper contrast ratios
Responsive: Desktop-first, mobile-friendly (min 375px viewport)
Accessibility: Must follow WCAG 2.2 AA accessibility standards
Component States: Must include default, hover, active, disabled, empty, error, and loading states
Consistency: Must match existing outbound email screens in visual language and interaction patterns

Section 10

Dependencies & Assumptions

Dependencies

  • AWS SES inbound receiving enabled for eu-west-1 region
  • S3 bucket provisioned with appropriate IAM policies for raw MIME and attachment storage
  • SNS topic configured with HTTPS subscription to the EaaS WebhookProcessor endpoint
  • MX records manageable by customers at their DNS provider (Cloudflare, Route53, GoDaddy, Namecheap, etc.)
  • MimeKit library supports all required MIME parsing operations (already validated in Sprint 3 spike)

Assumptions

Assumption Value Impact if Wrong
Average inbound email size (including attachments) 500 KB S3 storage costs, processing time
Peak inbound volume per tenant (launch estimate) 1,000 emails/hour Scaling architecture, queue sizing
Plain text replies (percentage of total inbound) 80% (< 50 KB each) Processing time estimates
Emails with attachments (percentage of total inbound) 20% (avg 2 attachments, 2 MB total) S3 storage projections
Customer webhook endpoint response time (P95) < 5 seconds Webhook timeout configuration, retry scheduling
